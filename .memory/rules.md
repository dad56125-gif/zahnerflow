# 记忆规则

`.memory/` 是本仓库的项目级记忆，不是流水账。它的作用是让后续修改持续对齐当前架构。

## 语言规则

1. `.memory/` 内所有正文必须使用中文。
2. 只有代码标识符、文件路径、命令、接口名、事件名、协议名、库名、专有名词可以保留英文。
3. 如果引用英文错误信息或命令输出，应尽量用中文解释其含义。
4. 新增或修改 `.memory/` 文件时，必须先检查是否混入无必要英文正文。

## 文件分工

- `rules.md`：更新纪律、锚点格式和强制规则。
- `design.md`：当前设计真相。如果代码与本文冲突，先检查代码，再修正文档。
- `changelog.md`：按设计锚点索引的演进历史。它记录重要变化为什么发生、改了什么、对设计造成什么影响。

## 锚点系统

所有可能被长期引用的架构规则、模块或流程，都必须在 `design.md` 中拥有稳定锚点。

锚点格式：

```text
[领域-主题]
```

示例：

- `[运行时-单进程]`
- `[运行时-AppRuntime]`
- `[设备-连接路由]`
- `[设备-真机驱动]`
- `[执行-状态机]`
- `[数据-SQLite]`
- `[接口-前端契约]`
- `[启动-单端口]`

`changelog.md` 必须使用与 `design.md` 完全一致的锚点文本。

## 更新规则

1. 如果修改影响架构、运行拓扑、数据流、接口契约、设备行为、启动行为或持久化方式，必须同时更新 `design.md` 和 `changelog.md`。
2. 如果只是局部缺陷修复且不改变设计，只有当原因或影响会影响后续判断时，才需要更新 `changelog.md`。
3. 如果修改让 `design.md` 里的旧描述失效，必须直接修正旧描述。不要追加一段相互矛盾的新说明后留下僵尸文档。
4. `design.md` 描述当前系统，不描述愿景。未来计划只能写在明确标注的“未完成事项”里。
5. `changelog.md` 每条记录必须包含影响锚点、原因、变更、设计影响和验证证据。
6. 禁止在未记录性质的情况下引入兼容壳、过渡层或回退路径。必须说明它是最终设计还是临时债务。

## 变更记录格式

```text
## YYYY-MM-DD - 简短标题

锚点：[锚点一]，[锚点二]
原因：是什么约束、问题或决策触发了这次变化。
变更：代码或行为具体发生了什么变化。
设计影响：`design.md` 中现在必须成立的设计事实。
验证：命令、运行检查、测试或其他证据。
```

## 设计条目格式

每个设计锚点都应回答：

- 当前规则：现在必须成立的事实。
- 归属文件：规则由哪些文件实现。
- 允许变化：什么可以演进且不破坏该设计。
- 禁止事项：什么不能被重新引入。
- 最近复核：日期和原因。

## Agent 完成前检查

架构相关工作结束前，必须确认：

- `changelog.md` 中每个被触及的锚点都存在于 `design.md`。
- 每条有设计影响的记录，都能在 `design.md` 中找到对应的当前设计事实。
- 没有旧设计描述与新代码相互矛盾。
- 启动、运行时、端口、进程等声明都有命令、搜索结果或运行检查支撑。

## 前端代码质量检查

每次修改前端代码前，应检查以下违规项：

### Barrel 文件（index.ts 重导出）

- 禁止在 `apps/frontend/src/` 下新增 barrel 文件（`index.ts`）。
- 已有的 barrel 文件（如 `state/index.ts`、`workflow/index.ts`、`modules/*/index.ts`）如果仍在被 import 使用，应逐步迁移为直接 import 源文件路径后删除。
- 迁移方法：将 `import { X } from '../state'` 改为 `import { X } from '../state/canvasStore'`（指向实际源文件）。

### Console 调试语句

- 禁止在前端代码中保留 `console.log`、`console.debug`、`console.info` 等调试输出。
- 允许保留 `console.error` 和 `console.warn`，用于真正的错误和警告场景。
- 开发完成后的调试语句必须清理，不留残留。
