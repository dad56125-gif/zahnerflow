# 简化数据库与文件组织方案

## 设计目标
- 无鉴权，仅记录归属：用 `owner_name`、`individual_name` 便于检索和目录生成。
- 保存工作流、节点与参数；支持按工作流/节点类型“整体修改某参数”。
- 数据文件落盘到固定目录结构，数据库仅登记元数据/路径。

## 最小模型（4 张表）
- `workflow`：工作流主表（id, owner_name, individual_name, title, description, tags, created_at, updated_at）。
  - 约束：`UNIQUE(owner_name, individual_name, title)`；索引 `(owner_name, individual_name)`。
- `node`：节点（id, workflow_id, node_key, node_type, display_name, sort_order, enabled, position_json）。
  - 约束：`UNIQUE(workflow_id, node_key)`；索引 `(workflow_id, node_type)`。
- `node_param`：节点参数键值对（id, node_id, key, value_text, value_num, value_json, value_type, updated_at）。
  - 约束：`UNIQUE(node_id, key)`；索引 `(node_id)`。
- `data_file`：测量文件登记（id, owner_name, individual_name, test_type, prefix, cycle, ts, filename, rel_path, size, sha256, workflow_id?, node_id?）。
  - 约束：`UNIQUE(owner_name, individual_name, test_type, filename)`；索引 `(owner_name, individual_name, test_type)`。

## 文件目录规范
- 根：`archive/`
- 路径：`archive/[owner_name]/[individual_name]/[test_type]/[prefix]-[cycle]-[YYYYMMDD-HHMMSS].扩展名`
- 例：`archive/alice/Cell-01/eis/expA-005-20251004-153012.dta`

## 批量参数修改示例（SQL）
- 修改某工作流下所有节点的 `gain`：
  `UPDATE node_param SET value_num=5, value_type='number' WHERE key='gain' AND node_id IN (SELECT id FROM node WHERE workflow_id=$1);`
- 仅针对 `eis` 节点：
  `UPDATE node_param p SET value_num=5 FROM node n WHERE p.node_id=n.id AND n.workflow_id=$1 AND n.node_type='eis' AND p.key='gain';`

## 与现有后端对齐与改动建议
- 现状：`apps/backend` 使用 `WorkflowStorageService` 将工作流存为 `data/workflows/workflows.json`，接口模型 `Workflow/WorkflowDefinition` 不含归属信息。
- 建议的最小改动：
  - 在 `Workflow` 增加：`ownerName`、`individualName`（或在 `definition` 增加同名字段），并在 `create/update` 接口透传。
  - `WorkflowStorageService` 同步读写这两个字段，保持 JSON 存储不变。
  - 为“整体修改参数”新增服务方法：遍历 `definition.nodes[*].config`，批量更新指定 key；后续若迁移到数据库，再落到 `node/node_param`。
  - 新增“数据文件登记”服务（可先用 JSON 索引）：生成并返回标准化目录与文件名，写磁盘并记录 `rel_path`、`filename`。
- 可选：未来接入数据库时，按上表结构直接迁移，无需改动文件夹组织与 API 形态（仅由服务层改为 DB 持久化）。
