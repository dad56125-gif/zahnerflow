# 事件总线与日志管理协作架构

## 问题分析

当前系统中存在真正的架构冲突：

### 1. 事件订阅者的日志失控
**NotificationEventHandler**作为SimpleEventBus的核心订阅者，内部使用直接Logger调用：
```typescript
@Injectable()
export class NotificationEventHandler implements EventHandler {
  private readonly logger = new Logger(NotificationEventHandler.name); // 问题所在！

  constructor(
    private readonly eventBus: SimpleEventBus,
    private readonly workflowGateway: WorkflowGateway,
  ) {
    this.registerEventHandlers();
    this.logger.log(`Registered ${Object.keys(handlers).length} notification event handlers`); // 绕过ConsoleDisplayManager
  }

  async handle(event: EventPayload): Promise<void> {
    this.logger.debug(`Processing notification event: ${event.type}`); // 绕过ConsoleDisplayManager
    // ...
  }
}
```

### 2. 双重日志系统冲突
- **SimpleEventBus**: 事件传递机制
- **ConsoleDisplayManager**: 日志级别控制机制
- **NotificationEventHandler**: 作为桥梁，但使用直接Logger输出，完全绕过控制

## 核心问题

### 1. 日志控制失效
NotificationEventHandler中的日志输出：
- 第49行：注册统计信息日志
- 第57行：事件处理调试日志
- 第71行：无处理器警告日志
- 第74行：处理失败错误日志

**这些日志完全不受ConsoleDisplayManager控制！**

### 2. 架构设计缺陷
事件订阅者应该：
- 只处理业务逻辑
- 通过统一的日志系统输出
- 遵循全局的日志级别策略

但现实是：
- 每个订阅者都有自己的Logger实例
- 直接输出日志，绕过控制机制
- 造成日志管理的混乱

## 设计原则

### 单一职责原则
- **SimpleEventBus**: 专门处理业务逻辑事件传递
- **ConsoleDisplayManager**: 专门处理日志输出控制

### 关注点分离
- 业务逻辑不应该关心日志输出
- 日志系统不应该影响业务流程

## 协作方案

### 方案A: 完全分离（推荐）
```
业务逻辑 → SimpleEventBus → 订阅者 → ConsoleDisplayManager.log() → 输出
```

**优点:**
- 职责清晰
- 业务逻辑无需感知日志系统
- 可以独立扩展

**缺点:**
- 需要在所有订阅者中手动调用ConsoleDisplayManager

### 方案B: ConsoleDisplayManager包装器
```
业务逻辑 → ConsoleDisplayManager.log() → 内部可选触发事件
```

**优点:**
- 统一的日志接口
- 可以同时控制输出和事件

**缺点:**
- 职责混合
- 增加复杂性

### 方案C: 事件总线拦截器
```
业务逻辑 → SimpleEventBus → 拦截器 → ConsoleDisplayManager过滤 → 订阅者
```

**优点:**
- 不修改现有代码
- 集中控制

**缺点:**
- 影响性能
- 增加系统复杂度

## 推荐实现方案

### 1. 核心架构 - 修复事件订阅者
**关键修改：让所有事件订阅者都使用ConsoleDisplayManager**

```typescript
@Injectable()
export class NotificationEventHandler implements EventHandler {
  constructor(
    private readonly eventBus: SimpleEventBus,
    private readonly workflowGateway: WorkflowGateway,
    private readonly consoleManager: ConsoleDisplayManager // 注入ConsoleDisplayManager
  ) {
    this.registerEventHandlers();
    // 使用ConsoleDisplayManager而不是直接Logger
    this.consoleManager.log('NotificationEventHandler', 'enableLog',
      `Registered ${Object.keys(handlers).length} notification event handlers`);
  }

  async handle(event: EventPayload): Promise<void> {
    try {
      // 使用ConsoleDisplayManager控制调试日志
      this.consoleManager.log('NotificationEventHandler', 'enableDebug',
        `Processing notification event: ${event.type}`, {
          eventType: event.type,
          timestamp: event.timestamp,
          context: event.context
        });

      // 事件处理逻辑...
      const methodName = `handle${event.type.split('.').map(part =>
        part.charAt(0).toUpperCase() + part.slice(1)
      ).join('')}`;

      if (typeof (this as any)[methodName] === 'function') {
        await (this as any)[methodName](event);
      } else {
        // 使用ConsoleDisplayManager控制警告日志
        this.consoleManager.log('NotificationEventHandler', 'enableWarn',
          `No notification handler found for event type: ${event.type}`);
      }
    } catch (error) {
      // 使用ConsoleDisplayManager控制错误日志
      this.consoleManager.log('NotificationEventHandler', 'enableError',
        `Failed to handle notification event: ${event.type}`, error);
    }
  }
}
```

## 具体实现策略

### 1. 渐进式改造
- 第一步：在关键模块中注入ConsoleDisplayManager
- 第二步：逐步替换直接Logger调用
- 第三步：验证日志级别控制效果

### 2. 通知系统修改
```typescript
@Injectable()
export class NotificationService {
  constructor(
    private eventBus: SimpleEventBus,
    private consoleManager: ConsoleDisplayManager
  ) {}

  sendNotification(notification: any) {
    // 业务事件
    this.eventBus.emit('notification.sent', notification);

    // 日志输出
    this.consoleManager.log('NotificationService', 'enableLog',
      `Notification sent: ${notification.title}`);
  }
}
```

### 2. 集成测试

- 验证事件总线的处理效率

## 总结

推荐采用**完全分离**的协作模式：

1. **SimpleEventBus** 专注于业务逻辑事件传递
2. **ConsoleDisplayManager** 专注于日志输出控制
3. 通过依赖注入在需要的地方使用日志系统
4. 保持两个系统的独立性和可测试性

这种设计符合单一职责原则，既解决了当前问题，又为未来的扩展提供了良好的基础。
