1.0 概述
本工作流文档详细描述了一个测量任务从前端用户界面发起，直至底层设备硬件执行的完整数据流转路径。此流程基于一次错误排查（output_path 字段缺失）的分析结果，明确了数据在不同系统层级（前端、NestJS后端、Python设备接口）之间的结构转换和处理逻辑。
2.0 涉及的核心文件与组件
前端 (Frontend Application)
apps/frontend/src/nodes/types.ts: 定义节点的基础元数据和默认参数。
apps/frontend/src/nodes/eis-node.tsx: 负责渲染节点UI、管理和更新节点参数状态。
apps/frontend/src/App.tsx: 管理整个工作流的状态，并负责在执行时将前端节点数据序列化后发送至后端。
后端服务层 (NestJS Backend)
apps/backend/src/modules/execution/execution.service.ts: 工作流执行引擎，负责解析前端发送的节点数据，并根据节点类型调度相应的设备服务。
apps/backend/src/modules/zahner-zennium/zahner-zennium.service.ts: 具体的设备服务，封装了对Python设备接口的HTTP调用。
设备控制层 (Python Device Interface)
apps/backend/scripts/zahner_device.py: 提供与硬件直接交互的FastAPI端点，接收参数并执行物理测量。
3.0 数据流转执行序列
阶段一：前端 - 节点实例化与参数配置
节点定义加载: 当用户在前端画布上创建一个新的 "恒电位EIS" 节点时，系统从 apps/frontend/src/nodes/types.ts 文件中读取 eis_potentiostatic 节点的 defaultParameters 对象。此对象包含了所有必需的参数，包括 output_config 及其子字段 output_path。
UI组件渲染: apps/frontend/src/nodes/eis-node.tsx 组件使用这些默认参数来初始化其内部状态 (useState)，并渲染出参数配置表单。
参数更新: 用户通过UI界面修改参数。组件的事件处理函数 (handleParameterChange, handleOutputConfigChange) 更新组件状态，并将更新后的完整参数对象通过 onUpdate 回调函数同步至 App.tsx 中管理的全局节点状态。此时，节点数据结构为 { id: '...', type: '...', data: { parameters: { ... } } }。
阶段二：前端 - 工作流执行与数据封装
触发执行: 用户点击执行按钮，触发在 apps/frontend/src/App.tsx 中定义的执行函数。
数据结构转换: 该函数遍历当前工作流中的所有节点。对于每一个节点，它会创建一个新的、用于网络传输的简化对象。在此过程中，发生了一次关键的数据结构重塑：
源结构 (前端状态): node.data.parameters
目标结构 (发送至后端): node.config
前端将存储在 node.data.parameters 中的参数对象，赋值给一个名为 config 的新字段。
API请求发送: 前端将包含 { id, type, name, config } 结构节点数组的整个工作流定义，通过HTTP请求发送到NestJS后端的执行端点。
阶段三：后端 (NestJS) - 节点解析与服务调度
接收工作流: NestJS后端的控制器接收到来自前端的HTTP请求，并将工作流数据传递给 apps/backend/src/modules/execution/execution.service.ts。
节点调度: ExecutionService 的 executeNode 方法开始处理单个节点。它使用 switch 语句匹配 node.type ('eis_potentiostatic')。
参数提取: 执行服务从传入的节点对象中提取参数。根据排查后的修正逻辑，它现在从 node.config 字段（而非 node.data.parameters）读取完整的参数对象。
服务调用: ExecutionService 将提取出的参数对象 (node.config) 传递给 apps/backend/src/modules/zahner-zennium/zahner-zennium.service.ts 中的 executeEISPotentiostatic 方法。
阶段四：后端 (NestJS) - 接口调用封装
HTTP请求构建: ZahnerZenniumService 的 executeEISPotentiostatic 方法接收到参数对象。
请求转发: 该方法将此参数对象作为请求体（Body），向Python FastAPI设备控制层发起一个 POST /measure/eis/potentiostatic 的HTTP请求。
阶段五：设备控制层 (Python) - 参数验证与硬件执行
端点接收: apps/backend/scripts/zahner_device.py 中的 /measure/eis/potentiostatic FastAPI端点接收到来自NestJS服务的请求。
参数验证: FastAPI利用Pydantic数据模型 (EISPotentiostaticRequest) 对请求体进行自动解析和验证。它期望请求体的JSON结构与模型定义完全匹配，直接包含 output_config, eis_lower_frequency 等字段。由于NestJS已将 node.config 的内容直接作为请求体发送，数据结构此时是匹配的，验证通过。
原始错误原因: 错误的路径 (node.data.parameters) 导致 node.config 为空，进而使发送到此处的请求体为空，Pydantic验证失败并报告 output_path 字段缺失。
设备执行: 验证通过后，函数从解析后的请求对象中获取各项参数，并调用底层的 device_wrapper 方法来配置硬件并执行实际的EIS测量。
返回结果: 测量完成后，Python服务将结果封装成JSON格式，逐层返回给NestJS服务，最终传递回前端界面。
