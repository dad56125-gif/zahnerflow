# 保存文件方法讨论

## 问题
现有代码一次性保存数据，程序崩溃时全部丢失。

## 解决方案
每5分钟保存一次。

## 代码
```python
def _save_data_to_csv(filename, fieldnames, data):
    import csv, os
    with open(filename, 'a', newline='') as f:
        w = csv.DictWriter(f, fieldnames=fieldnames)
        if not os.path.exists(filename):
            w.writeheader()
        w.writerows(data)

# 测量循环修改
measurement_data = []
last_save_time = time.monotonic()

for _ in accurate_timer(duration=polarization_time, interval=sampling_time):
    current = device_wrapper.getCurrent()
    measurement_data.append({"time": elapsed_time, "current": current})

    # 每5分钟保存
    if time.monotonic() - last_save_time >= 300:
        _save_data_to_csv(output_file, ['time', 'current'], measurement_data)
        last_save_time = time.monotonic()
        measurement_data.clear()
        print("[自动保存] 已保存")

    # 异常直接退出
    if not (min_current <= current <= max_current):
        break

# 最终保存
if measurement_data:
    _save_data_to_csv(output_file, ['time', 'current'], measurement_data)
```

## 关键决策
- 异常时不保存：错误数据没有保留价值
- 不用临时文件：直接写入最终文件，数据完全一样
- 保存函数只写入：保持原始数据精度，不做处理

## 统计信息处理变更
**涉及文件**: zahner_device.py

**问题**: 新方案中measurement_data.clear()会清空内存，导致统计信息计算错误。

**解决方案**: 从CSV文件读取数据后计算统计信息，而不是从内存计算。这样保证数据准确性且实现简单。

**技术逻辑**:
1. 测量完成后从完整CSV文件读取数据
2. 计算统计信息（平均值、最小值、最大值、数据点数）
3. 返回准确的统计结果

## Files模块重构方案

**涉及文件**: files.service.ts

**问题**: 现有Files模块未使用，且路径结构不符合实验数据组织需求。需要重新设计为项目-样品-测试的三层结构。

**解决方案**: 重构接口设计，支持项目化数据管理，实现自动测试分类。

**接口设计**:
```typescript
export interface RegisterFilePayload {
  projectName: string;        // 项目名
  individualName: string;     // 样品编号/测试文件夹
  testType: string;          // 测试类型（eis/iv/ocp等）
  basePath?: string;         // 用户指定基础路径，默认'archive'
  filename: string;          // 设备层生成的文件名
}
```

**设计思路**: 建立以项目为核心的数据组织体系，支持多样品、多测试的复杂实验场景。

**关键决策**:
- projectName替换ownerName，建立项目概念
- individualName作为样品编号，支持多样品管理
- testType自动分类到对应子文件夹
- 用户可选择基础路径，增强灵活性

**技术逻辑**:
1. 接收项目名、样品名、测试类型和基础路径
2. 生成结构化文件夹路径：basePath/{projectName}/{individualName}/{testType}/
3. 直接使用设备层生成的文件名，避免重复命名
4. 支持测试类型自动分类映射

**文件夹结构示例**:
```
basePath/
├─ 项目A/
│   ├─ sample001/
│   │   ├─ eis/
│   │   └─ iv/
│   └─ sample002/
└─ 项目B/
```

## Files模块与设备API连接方案

**涉及文件**: 新增measurement.controller.ts，修改zahner_device.py中8个测量函数

**问题**: Files模块未使用，需要建立后端控制路径，设备层专注测量的架构连接。

**解决方案**: 后端通过Files服务获取标准化目录路径，传递给设备API，设备API使用该路径保存文件。

**设计思路**: 建立清晰的职责分离，后端统一管理文件路径，设备层专注测量执行。

**关键决策**:
- Files服务只返回目录路径，不处理文件名
- 设备API使用后端传递的目录路径保存文件
- 后端控制器作为协调者，连接Files服务和设备API
- 文件路径管理集中在后端

**技术逻辑**:
1. 前端发送测量请求到后端控制器
2. 后端调用Files服务获取标准化目录路径
3. 后端将目录路径添加到设备API参数中
4. 设备API在指定目录下保存数据
5. 返回测量结果和文件信息

**接口设计**:
```typescript
// Files服务返回
interface RegisterFileResult {
  dirPath: string;  // 目录路径
}

// 后端控制器接收
interface MeasurementRequest {
  projectName: string;     // 项目名
  individualName: string;  // 样品编号
  testType: string;        // 测量类型
  basePath?: string;       // 用户指定基础路径，默认'C:\data\archive'
}

// 设备API参数扩展
interface DeviceMeasurementParams {
  final_path: string;  // Files服务提供的目录路径
  // 其他测量参数保持不变
}
```

**数据流架构**:
```
前端[projectName+individualName+testType+basePath] → 后端控制器 → Files服务 → 标准化目录 → 设备API → 文件保存
```

**设备API修改要点**:
- 移除路径创建和转换逻辑
- 新增final_path参数接收目录
- 使用final_path作为文件保存基础目录

**职责分工**:
- **Files服务**: 根据四个参数提供Windows标准化的目录结构
- **后端控制器**: 路径获取和参数传递
- **设备API**: 测量执行和文件保存

## 用户系统设计

### 问题描述

现有系统缺少用户层级管理，无法按用户分类和隔离数据。工作流和文件管理仅支持基本的归属字段，缺乏用户系统支撑。

### 解决方案核心要点

- **无鉴权简化设计**：仅用于数据分类，不实现登录验证
- **独立用户模块**：`users`模块独立于业务逻辑，提供CRUD操作
- **最小化字段设计**：用户表仅存储user标识、邮箱、创建时间
- **数据关联**：通过`user`字段关联工作流和文件路径表

### 设计思路

**数据表结构**：
- 新增`users`表存储用户基本信息
- 独立`users`模块（UsersModule + UsersService + UsersController）
- 后端提供RESTful API管理用户

**关键决策**：
1. `user`字段双重职责：登录标识 + 姓名显示
2. 无修改用户方法：基于KISS原则，最小化实现
3. 软删除机制：删除用户时保留其创建的工作流和文件路径记录

### 技术逻辑

**用户创建流程**：
1. 接收POST `/users`请求
2. UsersController接收参数并调用UsersService
3. UsersService调用DbService.createUser()
4. DbService生成唯一ID并存入users数组
5. 返回成功/失败状态

**用户删除流程**：
1. 接收DELETE `/users/{user}`请求
2. 从users数组中移除记录
3. 注意：不级联删除工作流和文件路径记录

### 涉及修改的文件范围

**新增文件**：
- `apps/backend/src/modules/users/users.service.ts`: 用户业务逻辑
- `apps/backend/src/modules/users/users.controller.ts`: HTTP接口
- `apps/backend/src/modules/users/users.module.ts`: 模块定义

**修改文件**：
- `apps/backend/src/db/db.service.ts`: 扩展users数据模型和CRUD方法
- `apps/backend/src/app.module.ts`: 注册UsersModule

### 定义的接口类型

**用户管理接口**：
```typescript
// 创建用户
POST /api/users
Request: { user: string; email?: string }
Response: { success: boolean; message: string }

// 查询用户列表
GET /api/users
Response: { users: string[] }

// 删除用户
DELETE /api/users/{user}
Response: { success: boolean; message: string }
```

### 数据结构定义

**users表结构**：
```typescript
{
  id: string;                    // 用户唯一ID（格式：user_时间戳）
  user: string;                  // 用户标识（登录名，同时作为姓名）
  email: string | null;          // 邮箱（可空）
  created_at: string;            // 创建时间（ISO字符串）
}
```

**实施要点**：
- 用户名唯一性检查：创建时避免重复
- 前后端字段命名一致性：全部使用snake_case
- 默认用户初始化：系统启动时创建默认用户

## 文件路径管理实现

### 问题描述

现有系统缺少文件路径统一配置入口。用户需要在测量前设置基础路径、项目名和样品编号，但当前这些参数散落在各个节点中，缺乏集中管理。

### 解决方案核心要点

- **统一入口**：在Toolbar中间添加"文件路径"按钮
- **Overlay面板**：点击按钮打开全屏配置面板
- **三项配置**：基础路径、项目名、样品编号
- **简化逻辑**：测试类型自动匹配，无需手动选择

### 设计思路

**Toolbar按钮位置**：
在Toolbar组件左侧（新建/打开/保存）和右侧（运行/停止/工作流）之间添加中间区域控制按钮。

**按钮样式**：
- 使用`btn-secondary`样式，无状态区分
- 图标：📁，文字标签："文件路径"
- 按钮点击即打开面板

**配置面板界面**：

```
+---------------------------------+
|         文件路径配置             |
|---------------------------------|
| 基础路径: [输入框] [📁浏览]     |
|                                 |
| 项目名:   [下拉选择&输入框]   |
|                                 |
| 样品编号: [输入框]              |
|                                 |
|        [取消]    [确定]         |
+---------------------------------+
```

### 关键决策

1. **按钮状态**：移除激活/未激活区分，统一使用次要按钮样式
2. **项目名设计**：下拉选择已有项目 + 直接输入创建新项目
3. **基础路径**：文本输入 + Windows文件浏览器选择
4. **测试类型**：后端自动根据测量节点类型匹配，前端无需选择
5. **用户选择来源**：用户在TopNavbar下拉栏选择，选择后文件路径配置根据当前用户自动匹配已有配置

### 技术逻辑

**调用链**：
Toolbar文件路径按钮 → App.tsx状态管理 → Canvas.tsx overlay渲染 → FilePathManagerUI组件

**配置交互**：
- 基础路径：支持手动输入或📁按钮调用Windows文件选择器
- 项目名：下拉已有项目列表 + 输入框创建新项目
- 样品编号：纯文本输入

**数据流**：
- 用户在TopNavbar下拉选择 → 设置当前用户
- 用户配置路径参数 → 根据当前用户自动匹配已有配置 → 后端数据库存储（data_file_path表）
- 测量节点执行时读取路径 → 后端自动匹配测试类型（eis/ocp/iv等） → 文件保存

**用户关联逻辑**：
- TopNavbar用户下拉选择后，设置全局当前用户
- 文件路径配置面板加载时，根据当前用户查询已有配置
- 匹配规则：user + project_name + individual_name
- 保存配置时，自动设置user字段为当前选择用户
