# 1.0.0 复盘洞察

## 被验证有效的方案

1. 单进程运行时降低了设备状态、执行状态和错误解释的复杂度。
2. 后端展开事实统一服务 ETA、执行、进度、报告和展开预览，避免了前端和后端步骤索引不一致。
3. fingerprint 自动归档比保存式 workflow CRUD 更适合实验记录追溯。
4. 实验记录按 workflow 组织，比平铺 execution 更符合用户找回实验的方式。
5. 后端运行前元数据检查能避免空用户、空项目和空样品名污染实验记录。
6. Furnace 历史概览改为聚合查询后，避免了打开 modal 即加载大范围原始点的性能问题。
7. BEM、按钮、输入、浮层和通信入口统一后，前端回归面更可控。

## 被废弃方案的原因

1. 多服务和 worker IPC：对本地单用户程序来说状态边界过重。
2. 前端估算运行时间：无法掌握设备事实，只能作为猜测。
3. 保存式 workflow 管理：会破坏实验定义不可变性和报告追溯。
4. 手动启动/停止节点：容易遗漏，且把仪器生命周期责任推给用户。
5. 最近一次内存报告：无法支撑历史报告和失败后完整流程复盘。
6. 表格优先的 Furnace 历史页：信息密度低，性能压力大。
7. 旧类名兼容层：会让迁移长期半完成，增加样式断链风险。

## 留下的技术债

1. Sass 仍使用 `@import`，后续应迁移到 `@use` / `@forward`。
2. 前端构建产物有大 chunk，需要按路由或重模块拆分。
3. 部分 TSX 仍存在 inline style，需要继续收敛到 SCSS 和设计令牌。
4. package metadata 中版本号不完全一致，需要建立发布前版本检查。
5. 执行级环境快照缺失限制了报告解释能力。
6. 长期历史数据还没有降采样和保留策略。
7. BEM 审计还有动态类和工具类候选需要持续过滤和治理。

## 留下的交互债

1. 工作流块不支持嵌套和块内编辑，用户复杂组合需求需要新版本重新设计。
2. ETA 只有来源类型，没有置信度或偏差解释。
3. 实验地图是探索入口，后续仍需要更好的筛选、聚类和定位反馈。
4. 运行前元数据确认是局部 5 秒窗口，后续可评估更明确的确认交互。
5. 报告仍偏事实罗列，缺少基于执行级快照的解释性摘要。

## 对后续版本的启发

1. 新功能必须先确认事实源归属：后端事实、前端展示、SQLite 持久化或设备反馈，不能混用。
2. 用户可见的历史与报告必须优先保护追溯性，不应为了编辑便利覆盖旧定义。
3. 本地应用也需要清晰信任边界：串口占用、外部软件、设备内部不可中断测试都要用事实解释。
4. “看起来可用但无写入路径”的接口比没有接口更危险，应及时删除或标记历史。
5. 文档需要区分当前设计、历史变更、审计快照和后续待办，不能混在同一段落里。
