# 1.0.0 关键决策

## 1. 本地单进程优先

决策：当前后端是单 Python 进程，`AppRuntime` 是进程内唯一协调器。

原因：ZahnerFlow 1.0.0 是本地、单用户、仪器控制程序。多服务、worker IPC 和设备服务端口会增加状态恢复、设备连接和错误解释的复杂度。

结果：启动、设备连接、执行状态、ETA、SQLite 写入和 Socket.IO 快照都围绕一个运行时组织。

## 2. 后端事实优先

决策：执行、ETA、循环展开、工作流块展开、自动边界和报告明细都以后端展开结果为准。

原因：前端无法可靠知道设备真实参数、运行时状态、当前炉温、设备内部测试耗时和执行失败后的事实序列。

结果：前端负责展示和交互，不能维护并行事实源。

## 3. 工作流不可变定义

决策：工作流身份由后端 fingerprint 决定，名称、收藏、备注和来源只作为展示元数据。

原因：用户会从历史工作流加载、修改参数、再次运行。如果继续原地覆盖旧定义，实验报告无法追溯真实执行配置。

结果：相同定义复用同一 workflow；不同定义自动归档为新 workflow；实验记录负责找回和重命名。

## 4. 组合能力保守落地

决策：`workflow_block` v1 只引用已归档 workflow，支持展开/收缩，但不支持块内编辑器和嵌套块。

原因：块内编辑和嵌套会把执行展开、fingerprint、报告归属和 UI 编辑语义同时复杂化。

结果：v1 得到可用组合能力，同时保持执行事实和归档身份可解释。

## 5. 仪器生命周期自动化

决策：`startup` / `shutdown` 不再是普通用户节点，而由后端围绕测量节点自动插入。

原因：仪器启动和停止是测量工作流的边界行为，不应要求用户手动放置并承担遗漏风险。

结果：画布更接近实验意图，执行明细仍保留自动边界事实。

## 6. 报告只展示可追溯事实

决策：报告从 SQLite 执行历史和 step result 构建，不写没有执行级关联的环境因果解释。

原因：Furnace 和 MFC 采样在 1.0.0 中没有 execution 级强关联，不能把时间窗数据伪造成确定属于某次执行。

结果：报告可信边界清晰，后续如果要增强解释能力，必须先补执行级环境快照。

## 7. 前端基础设施收敛

决策：前端通信、浮层、按钮、输入、下拉、BEM 类名和组件目录都建立统一契约。

原因：早期 UI 和通信代码存在多入口、多命名和局部样式体系，导致回归定位困难。

结果：后续开发应复用既有 class contract 和 `runtimeClient.ts`，不再新建并行基础设施。
