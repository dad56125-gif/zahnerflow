# ZahnerFlow Agent Guide

本文件是 ZahnerFlow 仓库内的项目级代理约束，适用于 Codex 及其他代码代理。
它只描述本仓库的工作规则，不描述你的全局个人偏好。

## 1. 任务开始前必读

每次开始任务前，先读取以下文件：

- `.memory/rules.md`
- `.memory/design.md`

如果任务需要追溯既有设计决策、判断相近改动边界或核对某项设计为何形成，再按相关设计锚点检索 `.memory/changelog.md`；不需要在任务开始时全文读取。

`GEMINI.md` 只在需要兼容旧代理入口时参考。

## 2. Python 与环境

- 使用 `uv` 管理 Python 环境、依赖和命令执行。
- 不要为本项目切换到其他 Python 包管理方式。
- 运行 Python 相关命令时，优先使用 `uv run ...`。
- Git 提交信息必须使用中文。

### Git 与仓库内容

- 每次代码、文档、配置或项目规则发生变更并完成后，必须创建 Git 提交。
- 测试源码、测试配置、测试用例、测试夹具及测试目录不得纳入仓库或提交。
- 验证过程中产生的临时测试文件必须放在仓库外或使用已忽略的临时目录。

## 3. 当前项目架构事实

- 当前后端是单进程 Python 运行时，不是多服务架构。
- `AppRuntime` 是进程内唯一运行时协调器。
- SQLite 是本地持久化边界。
- 真机驱动由 Python 类直接导入调用，不作为独立服务启动。
- 禁止重新引入旧 worker IPC、额外设备服务端口或遗留 NestJS 后端作为当前运行时一部分。

如需精确表述，以 `.memory/design.md` 为准。

## 4. 修改代码时的强制规则

- 先核对真实代码、真实路由、真实端口、真实返回值，再做判断。
- 不要根据旧备忘、想象中的架构或名义分层直接下结论。
- 前端与后端契约变更后，先同步共享 contracts，再检查类型与构建是否通过。
- 前端通信入口以 `apps/frontend/src/runtimeClient.ts` 为准，不要重新分叉新的通信主路径。
- 如无明确设计记录，禁止加入兼容壳、过渡层或回退路径。

## 5. 验证约定

- 前端契约相关修改后，优先验证 shared contracts、`packages/types` 和前端构建。
- 后端测试从 `apps/python_backend` 上下文运行，必要时使用 `PYTHONPATH=.`。
- 与运行状态相关的问题，优先检查真实端口、真实快照、真实接口载荷。

## 6. 文档同步规则

如果修改影响以下任一内容，必须同步更新 `.memory/design.md`，并在 `.memory/changelog.md` 追加一条精简的修改记录：

- 架构
- 运行拓扑
- 数据流
- 接口契约
- 设备行为
- 启动方式
- 持久化方式

要求：

- `changelog.md` 中使用的锚点必须存在于 `design.md`
- 不需要预读整个 `changelog.md`；需要历史上下文时，只查询与当前锚点或改动主题相关的记录
- 变更完成后必须追加记录，说明原因、变更、当前设计影响和验证证据
- 不要保留与当前代码矛盾的旧描述

## 7. 项目内优先级

当多份说明发生冲突时，按以下顺序处理：

1. 当前代码与运行事实
2. `.memory/design.md`
3. `.memory/rules.md`
4. 本文件 `AGENTS.md`
5. `GEMINI.md`

如果仍有冲突，不要猜测，先以代码和运行结果为准，再修正文档。

## 8. 版本号管理约束

### 8.1 单一版本源

项目根目录的 `VERSION` 是应用版本号的唯一真实来源，格式为语义化版本 `MAJOR.MINOR.PATCH`，例如 `1.0.1`。根 `package.json`、`apps/frontend/package.json`、`apps/desktop/package.json`、`packages/types/package.json`、`pyproject.toml`、后端 `version.py` 和前端/桌面生成版本文件不得分别手动维护，必须由 `pnpm version:sync` 同步。

数据库 `schema_version`、API 主版本和应用版本必须分开管理，不能用提高应用版本代替数据库迁移或 API 版本变更。

### 8.2 升级规则

- `PATCH`：Bug 修复、UI 小调整、性能优化、日志改进和兼容性修复，不改变原有使用方式。
- `MINOR`：新增用户可见功能、页面、接口、设备能力或重要工作流改进，保持基本兼容。
- `MAJOR`：破坏性兼容变化、核心架构替换、不可向后兼容的数据迁移或正式稳定版发布。

在 `1.0.0` 之前，破坏性变化通常升级 `MINOR`。仅修改注释/开发文档、测试、格式化或尚未可发布的中间修改，通常不升级应用版本。无法确定时选择较小的升级类型并在最终总结中说明。

### 8.3 Codex 完成任务要求

完成可独立发布的变更前，必须判断升级类型；需要升级时先修改 `VERSION`，执行 `pnpm version:sync`，更新根 `CHANGELOG.md` 的对应版本章节，再执行 `pnpm version:check`。构建和发布脚本必须先通过版本检查；版本检查失败不得继续构建。

最终总结必须声明：原版本、新版本、升级类型和升级原因。

### 8.4 Definition of Done

- [ ] 功能或修复已完成
- [ ] 已运行相关测试
- [ ] 已判断版本升级类型
- [ ] 已更新 `VERSION`
- [ ] 已同步所有版本字段
- [ ] 已更新 `CHANGELOG.md`
- [ ] 数据库变化包含迁移
- [ ] 最终总结声明版本变化

禁止在多个文件中维护互不关联的版本号；禁止只修改 `package.json`；禁止修改数据库结构却不提供迁移；禁止用日期、构建次数或 Git 提交数替代应用版本号。
