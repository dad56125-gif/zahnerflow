
这是基于我们最终对齐的**“执行层主导、设备层纯粹、单用户串行”**的完整架构方案。

### 1. 核心设计原则

- 
- **物理与逻辑分离**：Python 设备层只产出物理数据（电压/电流/时间），完全不知道“工作流”、“节点”的存在。
- **执行层主导**：只有 ExecutionService 拥有上帝视角，负责将物理数据映射到具体的执行步骤上。
- **EIS 文件化**：EIS 测试不走流，保持原有的文件生成逻辑；只有 DC 类测试（OCP, Ramp, Chrono）走实时流。

------



### 2. 详细实施方案 (从后往前)

#### 第一层：Python 设备层 (生产者)

**角色**：纯粹的硬件驱动，无状态，只管吐数据。

1. 

2. **建立通道**：在 FastAPI 中增加一个 /ws WebSocket 端点，用于向外广播。

3. **同步桥接**：由于测量循环是同步阻塞的（while/for 循环），需要实现一个“同步调异步”的辅助机制（利用 asyncio 的线程安全方法），将数据从测量线程“发射”到 WebSocket 循环中。

4. **数据埋点**：

   - 

   - 仅在 **DC 类测试**（measure_ocp, measure_chrono..., measure_ramp...）的循环内部埋点。

   - **发送格式**：极致纯净，无任何 ID。

     codeJSON

     

     ```
     { "t": 1.25, "v": 0.500, "i": 0.002 }
     ```

   - **EIS 处理**：EIS 测量函数中**不**调用广播，保持静默。

#### 第二层：NestJS 设备服务层 (ZahnerService - 哑管道)

**角色**：数据搬运工，负责协议转换 (WS -> EventBus)，不包含业务逻辑。

1. 
2. **连接**：模块启动时，作为 Client 连接 Python 的 /ws。
3. **转发**：
   - 
   - 收到 WebSocket 消息 -> JSON.parse。
   - 直接扔进系统内部总线 EventBus，事件名为 device.raw_stream。
   - **不**附加任何 NodeId 或 ExecutionId，它不配知道这些。

#### 第三层：NestJS 执行层 (ExecutionService - 大脑)

**角色**：数据加工厂，拥有上下文状态 (Context Owner)。

1. 

2. **监听**：订阅 device.raw_stream 事件。

3. **状态校验**：

   - 
   - 检查 _globalState.status 是否为 'running'。如果不跑，数据直接丢弃。
   - 检查 _globalState.currentStep 是否存在。

4. **数据富化 (Enrichment)**：

   - 

   - 从内存快照中取出 **executionId** (执行批次号) 和 **stepIndex** (当前步骤索引)。

   - 将物理数据打包成业务数据：

     codeJavaScript

     

     ```
     {
       executionId: "exec_12345",
       stepIndex: 2,  // 代表这是第3个节点的运行数据
       data: { t: 1.25, v: 0.500, i: 0.002 }
     }
     ```

5. **广播**：调用 WorkflowGateway 发送给前端。

#### 第四层：前端 React (消费者)

**角色**：按需渲染。

1. 
2. **订阅**：监听 Socket.IO 的 measurement-stream 频道。
3. **过滤 (核心逻辑)**：
   - 
   - 组件 props 传入了 nodeIndex (该组件在工作流中的位置)。
   - **判断**：收到的 packet.executionId 是否等于当前运行ID？且 packet.stepIndex 是否等于我的 nodeIndex？
4. **渲染**：
   - 
   - 如果匹配：将 {x: t, y: i} 或 {x: v, y: i} 推入图表数组，触发重绘。
   - 如果不匹配：忽略。

------



### 3. 数据流向总结

1. 
2. **Python**: broadcast({t, v, i})
   ⬇️ *(WebSocket)*
3. **ZahnerService**: 收到 -> EventBus.emit('device.raw', {t, v, i})
   ⬇️ *(EventBus)*
4. **ExecutionService**: 监听到 -> 加上上下文 -> Gateway.broadcast({execId, index, data})
   ⬇️ *(Socket.IO)*
5. **React**: 收到 -> if (index === myIndex) draw()

### 4. 异常与边界处理

- 

- **停止/取消**：

  - 
  - 用户点击停止 -> ExecutionService 将状态置为 idle 或 cancelled。
  - Python 可能还在发最后几个包 -> ExecutionService 里的校验逻辑 if (status === 'running') 会拦截并丢弃这些数据，前端不会画出多余的尾巴。

- **EIS 期间**：

  - 
  - Python 不推数据 -> ExecutionService 收不到事件 -> 前端收不到流。
  - 前端逻辑保持不变：EIS 节点只在收到 node.completed 后，通过 HTTP 请求去加载 CSV 文件显示最终结果。
